# 剧情分支与世界状态管理

## 核心问题

玩家的选择不仅影响剧情走向，还会改变世界状态（环境、角色），
这些状态变化必须持续反映到后续的剧情生成和资产生成中。

## 世界状态：两级追踪

### 显式状态（选项触发，写入图数据库）
- 由玩家选择直接触发的状态变更
- 系统明确知道改了什么
- 必须持久化到图数据库节点属性中

示例：
```
选择"摘花"     → Location:公园.花 = 不存在
选择"保护她"   → Character:她.身体状态 = 健康, 好感度 += 1
选择"没保护"   → Character:她.身体状态 = 受伤(左臂)
```

### 隐式状态（剧情推理，不入库）
- 剧情自然发展带来的细节变化
- 不写入数据库，由 LLM 根据剧情摘要自行推理
- 如"经历战斗后的街道废墟感"

## 状态如何影响生成

### 影响剧情生成
- LLM 生成后续剧情时，当前世界状态作为上下文注入
- 确保剧情逻辑自洽（花摘了就不会再描写赏花情节）

### 影响资产生成/查询
- 生成场景图时，查询场景节点当前状态 → 决定 prompt
- 生成角色立绘时，查询角色当前状态 → 决定是否叠加伤痕等效果
- 查询资产池时，状态作为索引维度之一
