# 整体架构设计

## 核心三层架构

### 一、解析层：图数据库 + 文件系统

**图数据库**负责存储结构化关系数据：
- `Character` 节点 — 角色信息，关联资产文件夹（立绘、配音样本）
- `Location` 节点 — 场景/地点，关联资产文件夹（背景图、氛围音）
- `Event` 节点 — 剧情事件
- `Chapter` 节点 — 章节
- `WorldRule` 节点 — 世界观规则/设定

**边（关系）类型**：认识、敌对、同盟、发生在、导致、属于...

**文件系统**负责存储实际资产：
- 每个图节点关联一个文件夹路径
- 文件夹内存放该节点的所有资产文件（png、mp3、md 等）
- 图数据库只管关系和索引，资产管理与数据查询解耦

### 二、生成层：流式生成 + Gap 机制

**核心概念：Gap（预生成间隔）**
- 玩家当前在第 N 章，系统已预生成到第 N+Gap 章
- Gap 是系统性能和体验质量的核心调度指标

**Gap 策略**：
- Gap 过大 → 流畅但浪费资源（预生成的分支可能不被选择）
- Gap 过小 → 贴合选择但有缓冲风险
- 理想状态：Gap 动态调整

**生成流程**：
1. 先预生成前若干章供玩家开始游玩
2. 玩家推进过程中，根据已做选择持续生成后续章节
3. 生成内容包括：剧情文本、场景图、角色立绘、配音

**待定问题**：
- [ ] 分支预测策略：预生成所有分支 vs 只预生成最可能的分支？
- [ ] Gap 动态调整的具体策略

### 三、资产层：文件夹路径对应图节点

- 角色资产（立绘变体、表情、配音样本）存于角色节点关联文件夹
- 场景资产（背景图、氛围音效）存于场景节点关联文件夹
- 生成新资产时直接写入对应文件夹
- 图数据库记录资产索引和引用关系
