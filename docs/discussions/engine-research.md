# 游戏运行时引擎选型研究

> 研究时间：2026-04
> 需求背景：novel2gal 项目需要一个运行时引擎，核心特点是内容（剧情、图片、音频）由 AI 动态生成，非传统预制 VN。

## 核心需求清单

1. **运行时动态加载** — 文本、图片、音频均非预先硬编码，需运行时从文件系统/网络加载
2. **外部 API 集成** — 需调用 LLM（故事生成）、图像生成、TTS 等外部服务
3. **分支剧情** — 玩家选择驱动的分支叙事
4. **资产池系统** — 按需加载的资产索引与复用机制（角色×服装×情绪×程度）
5. **角色立绘变体** — 多情绪、多服装的动态立绘
6. **场景时间变体** — 同场景不同时间段的背景图
7. **跨平台** — Web、桌面、移动端

---

## 方案一：Ren'Py

**最新版本**：8.5.2 (2026-01-03)

### 优势
- **最大的 VN 生态系统** — VNDB 收录近 23,000 部 Ren'Py 作品，社区成熟，教程丰富
- **Python 底层** — 可嵌入任意 Python 代码，理论上可调用任何 Python 库
- **内置 VN 功能完善** — 存档/读档、回溯、自动播放、跳过等开箱即用
- **跨平台** — Windows、macOS、Linux、Android、iOS，HTML5（beta）
- **已有 ChatGPT 集成插件** — [RenPy-AutoScriptPlugin](https://github.com/Wendy-Nam/RenPy-AutoScriptPlugin)

### 劣势与限制
- **动态加载能力有限**
  - `renpy.image()` 可在运行时注册图片，`show expression` 可显示动态表达式
  - 但 Layered Image 系统的表达式在 init 时求值，不支持真正的运行时动态
  - 整体架构设计假设资产是预制的，动态加载是"逆流"使用
- **Web 端严重受限**
  - 不支持后台预加载图片（会导致帧率卡顿）
  - 不支持多线程（`renpy.invoke_in_thread()` 不可用）
  - 不支持网络请求（networking not supported in web builds）
  - 50MB+ 文件不被浏览器缓存
  - 动画表现受限
- **脚本驱动架构** — Ren'Py 的核心是 .rpy 脚本解释器，要实现"无脚本、纯动态"的流式内容需要大量 hack
- **API 集成** — 可通过 Python 调用 `requests`/`httpx`，但在 Web 端不可用；桌面端可行但需要处理异步问题（Ren'Py 主循环不是 async 友好的）

### 适用度评分：**3/10**

Ren'Py 为传统预制 VN 优化，动态生成场景是其设计意图之外的用法。桌面端勉强可行但别扭，Web 端基本不可行。

---

## 方案二：Web 引擎 — Monogatari.js

**状态**：活跃维护，MIT 开源，文档最近更新 2026-01

### 优势
- **Web 原生** — "你的游戏首先是一个网页"，HTML5/CSS3/JavaScript
- **PWA 支持** — 可安装为原生应用，支持离线播放
- **类 Ren'Py 语法** — 迁移成本低
- **响应式设计** — 自适应各种屏幕尺寸
- **MIT 开源** — 无商业限制

### 劣势
- **社区较小** — 相比 Ren'Py 用户基数小很多
- **动态加载能力** — 比 Ren'Py 好（JS 天然支持动态加载），但仍是面向预制脚本设计
- **功能有限** — 缺乏复杂的资产管理、层级图片合成等高级功能
- **维护节奏** — 主要由单人维护，更新频率不稳定

### 适用度评分：**4/10**

比 Ren'Py 在 Web 端更自然，但本质仍是预制脚本引擎，需要大量改造才能适配动态生成流程。

---

## 方案三：Web 引擎 — Pixi'VN（TypeScript + PixiJS）

**状态**：活跃开发中，npm 包 `@drincs/pixi-vn`

### 优势
- **现代 TypeScript 架构** — 类型安全，IDE 友好
- **PixiJS 2D 渲染** — 高性能 WebGL 渲染
- **UI 框架无关** — 不内置 UI，允许使用 React/Vue 构建界面，灵活度极高
- **模块化设计** — 对话、角色、画布、变量存储、存档等模块化提供
- **React 模板可用** — 官方提供 React + ViteJS 模板

### 劣势
- **社区较新** — 2024-2025 年起步，生态尚小
- **需要更多 DIY** — 不提供完整开箱即用体验，需要自行构建 UI 层
- **文档有限** — 相比成熟引擎文档不够全面

### 适用度评分：**6/10**

架构比传统 VN 引擎更适合动态场景——TypeScript + React + PixiJS 的组合天然支持动态加载和 API 调用。但仍需大量自定义开发。

---

## 方案四：Unity + Naninovel

**最新版本**：v1.20+，持续更新中

### 优势
- **最成熟的 Unity VN 框架** — 十年打磨，商业验证
- **开放 API 架构** — 每个内置功能都可通过 C# API 扩展/替换/覆写
- **运行时动态初始化** — `RuntimeInitializer.Initialize()` 支持动态启动
- **Addressable 资产系统** — 支持从远程主机加载资产，可注册自定义资源提供者
- **跨平台全覆盖** — Windows、macOS、Linux、Android、iOS、WebGL、主机
- **与 Unity 生态集成** — 可利用 Unity 的所有功能（网络、协程/async、着色器等）

### 劣势
- **商业授权** — 需购买许可证（Unity Asset Store 售价约 $150）
- **Unity 引擎体积** — Web 构建体积较大（最小约 30-50MB）
- **学习曲线** — 需要 Unity + C# 知识
- **过于重量级** — 对于纯 2D VN 来说，Unity 引擎本身是"大炮打蚊子"
- **Unity 商业政策风险** — 2023 年 Runtime Fee 事件后的信任问题（虽已撤回）

### 适用度评分：**5/10**

技术能力最强（动态加载、API 集成、跨平台都不是问题），但引擎过重，开发复杂度高，Web 端体积偏大。适合需要复杂 3D 表现或游戏机制的项目。

---

## 方案五：Godot + VN 插件

**Godot 版本**：4.3+
**主要插件**：Dialogic 2、Rakugo、DokiVN、Castelet

### 优势
- **开源免费** — MIT 许可证，无商业限制，无运行时费用
- **GDScript 简单** — 学习曲线低于 C#/Unity
- **跨平台** — Windows、macOS、Linux、Android、iOS、Web（HTML5）
- **轻量级** — 比 Unity 轻得多
- **Dialogic 2** — 较活跃的对话/VN 框架，Godot 4.3+ 支持

### 劣势
- **VN 生态不成熟** — 各插件分散，无统一标准，功能各有残缺
- **动态加载** — Godot 支持运行时资源加载（`load()`/`ResourceLoader`），但 VN 插件层面未专门设计
- **Web 端限制** — Godot HTML5 导出比 Unity 轻，但仍有限制
- **社区小众** — VN 开发者社区远小于 Ren'Py
- **API 集成** — 需要通过 GDScript 的 `HTTPRequest` 节点或 C# 互操作

### 适用度评分：**4/10**

比 Unity 轻量，比 Ren'Py 灵活，但 VN 生态不够成熟。如果选择 Godot，基本等于在 Godot 上自建 VN 系统。

## 方案六：自定义 Web 引擎（推荐方向）

**技术栈参考**：TypeScript + React/Vue + Canvas/WebGL

### 架构设想

```
┌─────────────────────────────────────────────┐
│  前端 UI 层（React/Vue）                       │
│  - 对话框、选项按钮、菜单、存档界面             │
│  - 角色立绘展示（CSS/Canvas 合成）             │
│  - 场景背景展示                               │
├─────────────────────────────────────────────┤
│  VN 运行时核心（TypeScript）                    │
│  - 场景状态机（当前场景、角色、对话队列）        │
│  - 资产加载器（从后端/文件系统动态拉取）         │
│  - 存档/读档管理                               │
│  - 选项/分支处理                               │
├─────────────────────────────────────────────┤
│  后端服务层（Node.js / Python）                 │
│  - LLM 调用（故事生成、对话生成）               │
│  - 图像生成调度（SD/Flux/DALL-E）              │
│  - TTS 调度（CosyVoice 等）                    │
│  - 图数据库查询（世界状态、角色关系）            │
│  - 资产池管理（索引、缓存、持久化）             │
└─────────────────────────────────────────────┘
```

### 优势
- **为动态生成量身定制** — 不需要绕过任何预制假设
- **原生 API 集成** — fetch/axios 直接调用外部服务，async/await 天然支持
- **资产动态加载无障碍** — Web 天然支持动态图片/音频加载
- **技术栈现代** — TypeScript 类型安全、React 组件化、生态庞大
- **跨平台**
  - Web：原生
  - 桌面：Electron / Tauri
  - 移动端：PWA / Capacitor / React Native
- **完全可控** — 资产池系统、Gap 预生成机制、世界状态管理全部按需设计
- **可渐进式开发** — 先实现核心对话循环，再逐步加入立绘合成、音效、TTS 等

### 劣势
- **开发工作量大** — 存档/读档、回溯、UI 转场等都要自己实现
- **无现成社区** — 出了问题只能自己解决
- **需要全栈能力** — 前端 + 后端 + AI 调度

### 可复用的开源组件
- **Pixi'VN** — 可参考其核心对话/状态管理模块
- **[aisimulator](https://github.com/greenlighttec/aisimulator)** — Next.js 上的 AI VN 实现，可参考架构
- **[react-visual-novel](https://github.com/utilfirst/react-visual-novel)** — React VN 库，可参考 UI 组件
- **PixiJS** — 如需高级 2D 渲染（立绘合成、特效）

### 适用度评分：**9/10**

对 novel2gal 项目来说是最佳选择。项目核心挑战（动态生成、AI 集成、资产池）不在"VN 引擎"层面，而在后端调度层面。前端 VN 展示逻辑本身并不复杂——本质上就是：显示背景 → 显示立绘 → 显示对话 → 等待选择 → 循环。自建引擎可以把精力集中在真正的难点上。

## 推荐结论

### 首选：自定义 Web 引擎（TypeScript + React）

**理由**：

1. **需求匹配度最高** — novel2gal 的核心是"AI 动态生成内容的运行时展示"，这与传统 VN 引擎"播放预制脚本"的设计目标根本不同。
2. **架构契合** — 项目已设计的三层架构（解析层 → 生成层 → 资产层）天然对应 Web 前后端分离架构。
3. **资产池系统** — `角色×服装×情绪×程度` 的索引和按需加载在 Web 后端中实现最自然。
4. **Gap 预生成机制** — 后端异步生成 + 前端流式消费的模式在 Web 架构中有成熟方案（WebSocket、SSE）。
5. **VN 前端并不复杂** — 对话框、立绘、背景、选项菜单的 UI 复杂度远低于后端的 AI 调度。用 React 组件化实现非常直接。

### 备选：Pixi'VN 作为基础

如果希望减少前端 VN 基础功能的开发量，可以基于 Pixi'VN 构建——它提供了对话管理、角色系统、存档等核心模块，同时不限制 UI 框架选择，TypeScript 类型安全也与项目技术方向一致。

### 不推荐：Ren'Py / Godot

尽管 Ren'Py 生态最大，但它的设计假设与本项目根本冲突。Godot 在 VN 领域生态不成熟，选择它等于在游戏引擎里重建 VN 系统，不如直接在 Web 上建。
